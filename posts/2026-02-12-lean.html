Kernel and prelude

There is lean kernel which knows about very few constructs. There is an elaborator which converts syntatic sugar to core the core constructs that lean knows about. As elaborator is working, it maintains an environment which contains all the constructs that are available in the program at some stage.

So, the kernel does not even know about Nat or Prop or Bool or even Type. These are the basic constructs that any lean programmar would want to be available. They are all defined in Init module which is automatically imported to all the lean programs. However, importing them in raw form (.lean file) would be performance killing. So, Init module has been compiled and the output of that compilation is some .olean files which are serialized snapshots of environment contributions. Note that the environment contributions of .olean files are not checked for correctness. They are assumed to be correct - since correctness had been checked when .olean files were getting built and we are assuming that they have not been tampered with.

Another thing is that for the purposes of theorem proving, we only "compile" the program in the sense of checking that all terms are well typed. We don't actually run the program.

Universes

There are some universes. Countably infinite universes. They are called Sort u for u = 0, 1, 2,... Prop is a short hand for Sort 0, Type n is a shorthand for Sort (n+1). Sort u lives in Sort (u+1)
Every term has a type. Each term "lives" in that type. Core lean kernel only has Sorts. In the init module we say that Type is same as Sort 0. And that Prop lives in Type, and that Nat lives in Type.

Inductive type

Define by giving various rules of constructing the type. 


inductive Color where
  | red
  | green
  | blue
So, the three given literals have the type Color. Color has not other values.

Critical part is that you can use recursion 


inductive List (α : Type) where
  | nil  : List α
  | cons : α → List α → List α

So, nil is type List α , cons 3 nil is of type List α, cons 2 (cons 3 nil) is of type List α, and cons 1 (cons 2 (cons 3 nil)) is of type List α.
The last thing is written in sugared form as [1, 2, 3]. How does this sugaring works - we will not dsicuss it here.
Similarly, Nat is also inductive type: 

inductive Nat where
  | zero : Nat
  | succ : Nat → Nat
So, Nat.zero, Nat.succ Nat.zero, Nat.succ Nat.succ Nat.zero etc are all of type Nat. Their canonical form is 0, 1, 2 etc. How the translation to and from the logical form (coming from the definition) to the literal form (0, 1, 2) happens is something ware not going to discuss. def andComm (a b : Prop) (ha : a) (hb : b) : And a b ↔ And b a :=
  Iff.intro
    (fun h => And.intro h.right h.left)
    (fun h => And.intro h.right h.left)
This is constructive. To produce a term of type And a b ↔ And b a, we need to do two things. Firsly, given a term of type And a b, we need to produce a term of type And b a. And secondly, vice versa. To to the firs thing, we write
(fun h => And.intro h.right h.left)

Class

We have Nat, and we have Nat.add. Let's not worry about how it is implemented.
We also have Float, and we have Float.add. Again, let's not worry about how it is implemented.
We also have Int, and we have Int.add.
We would like to use + polymorphically - when we say a + b, we want lean to figure out the types of a and b (check that they are same) and apply suitable add. We achieve this using class. 
class Add (α : Type u) where
  add : α → α → α
We register instances 
instance : Add Nat where
  add := Nat.add

instance : Add Float where
  add := Float.add

instance : Add Int where
  add := Int.add
and associated + with Add:
notation a + " " + b => Add.add a b
Actually it is not way it is done, they introduce an HAdd (heteroegnous add) to allow addition across types.

Pi Type

Finally we turn to the last of the three big topics - pi types. We have type constructors like myVector which take in an integer and produce a type which is dependent on that input - like myVector can give you myVector Nat 4 or myVector Float 5. However, as of now we don't have a mechanism to write a function that can produce values of different types.
E.g. let's say we want to write a function that takes an input n and output myVector Nat n with all values as 42. We will write it in one of the two ways: 

def allFortyTwo : (n: Nat) → myVector Nat n 
   | 0 => myVector.nil
   | n + 1 => myVector.cons 42 (allFortyTwo n)

def allFortyTwo (n : Nat) : myVector Nat n :=
  match n with
  | 0     => myVector.nil
  | n + 1 => myVector.cons 42 (allFortyTwo n)

Types of types in functions

A function take as input - explicit type: def f(x : Nat): Nat := x + 1


Params can also be explicit , like here
def g {t : Type} (x : t) : t := x
In this example, t will be inferred automatically.
However, following would not compile: def g {t : Type} (x : t) : t := x + 1
because there is not guarantee that the type you pass would support +.
So, for polymorphic increment you can have instance explicit type:
def g {t : Type} [Add t] [OfNat t 1] (x : t) : t := x + 1
which says that t supports an Add operation and [OfNat t 1] says literation 1 must make sense for type t.
We could also write
def g {t : Type} [Num t] (x : t) : t := x + 1
What really is [Add t] or [OfNat t 1] or [Num t] - we have handwaved it. We will clarify this later.
And then a function's type can be dependent type - where the return type depends on the input value.
(a: A) → B a where B is a function of type A → Type.
Eg. of a function with dependent type: 

def zeros : (n : Nat) → Vec Int n
  | 0     => []
  | n + 1 => 0 :: zeros n
Type of this function is dependent on the actual value of input n.

Next three lines say that there are two arguments to the only constructor (structure has only one constructor) First argument is val, and second argument is isLt which is of type LT.lt val n. LT.lt
